protein_sets

A single row contract for selecting, subsetting, slicing, and transforming human protein sequences by annotation.

python reviewed human proteome · 20,416 proteins auto-downloading cache MIT-style research tool

Every query — a named set, an annotation region, a synthetic-variant transform, or a bespoke study — returns a DataFrame on the same (item_id, uniprot, sequence, …) contract, so the output of one layer flows straight into the next and into downstream embedding / structure tools.

protein_sets is the canonical lens for any question about subsets, annotations, or categories of proteins, defaulting to the reviewed human UniProt proteome. The core idea is simple: pick a subset of proteins by whatever annotation you care about — secreted, transmembrane, has a signal peptide, is a transcription factor, lives at the cell surface — and then get back exactly the shape you need from it:

  • just the IDs — a list of UniProt accessions;
  • metadata in tabular form — pick any columns (gene, length, topology, GO, keywords…) as a DataFrame;
  • the full sequences — as a DataFrame or written straight to FASTA;
  • sub-sequences — slice out regions within those proteins (Pfam domains, signal-peptide spans, sites) with coordinates and the exact sliced string;
  • mutant forms — generate synthetic variants of the chosen proteins (e.g. a window mutation scan) on the same contract.

Those five outputs map onto the package's layers. It unifies four kinds of thing behind one interface: named sets (secretome, transmembrane, signal-peptide, surfaceome, transcription factors…), regions (Pfam domains, signal-peptide spans, sites — sub-sequences within proteins), transforms (mutation scans and other synthetic-sequence experiments), and studies (your own bespoke per-project datasets). Whatever you pick — a named set, a region, a transform, a study — comes back on one row contract (item_id, uniprot, sequence, …), so the output of one layer feeds straight into the next and into downstream embedding / structure tools.

Batteries included, but extensible. The named sets, region builders and transforms above ship with the package and run against auto-downloading reference data (UniProt, HPA, CSPA, humantfs, UniRef) — no manual file wrangling, every later call is instant. Two things you bring yourself: a new named set is a few lines registered in the package, and a study is a bespoke per-project dataset you drop under studies/<name>/ — once registered it resolves and slices exactly like the built-in sets.

Named sets built-in + extensible

Logically-defined filters inside a universe: secretome(), transmem(), signal_peptide(), set algebra and UniRef dedup. A dozen ship by default; add your own in a few lines.

Regions built-in

Sub-sequences within proteins: Pfam domains, signal-peptide spans, sites — with coordinates and the sliced sequence.

Transforms built-in

Synthetic sequences derived from real proteins: window mutation scans today, more to come.

Studies bring your own

Drop a bespoke per-project dataset under studies/<name>/ and it resolves and slices like any named set.

$Install

pip install git+https://github.com/jlaffy/protein_sets

Then:

from protein_sets import load_proteome, signal_peptide, secretome, resolve

load_proteome()          # full reviewed-human UniProt table (auto-downloads once)
signal_peptide()         # proteins with a signal-peptide annotation
secretome()              # secreted proteins ∩ reviewed human
resolve("secretome")     # → list of UniProt accessions
Data root & auto-download. On first use the package fetches what it needs (UniProt, HPA, CSPA, humantfs, UniRef) into a local cache; every later call is instant, and a 30-day TTL refreshes stale data automatically. The cache location is resolved once at import, in priority order: $PROTEIN_SETS_DATA → /ewsc/jlaffy/protein_sets (shared NFS cache, if present) → ~/.cache/protein_sets (fresh-install default). InterPro-backed domain regions currently need a local InterPro file (available on-cluster; auto-download planned for 2.0); everything else works out-of-the-box.

↓One contract, many outputs

Each query can be returned three ways, whichever the downstream tool needs:

DataFrame

The canonical output, carrying df.attrs['provenance'] for full lineage.

IDs

resolve(query) returns a plain list of UniProt accessions.

FASTA

to_fasta(df, path) writes records for non-Python tools.
